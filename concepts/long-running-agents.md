## 构建长时间运行 Agent 的有效 Harness

> 原文: Effective Harnesses for Long-Running Agents

> 作者: Justin Young（Anthropic 工程团队）

> 发布日期: 2025-11-26

> 翻译日期: 2026-04-14

---

### 概述

随着 AI Agent 变得越来越强大，开发者越来越希望让它们承担需要数小时甚至数天才能完成的复杂任务。然而，让 Agent 在多个 Context Window（上下文窗口）之间保持持续进展，仍然是一个开放性问题。

> 💡 术语解释 - Context Window（上下文窗口）:

> LLM 在一次对话中能处理的最大 token 数量。可以理解为 Agent 的「短期记忆容量」。当对话超过这个限制时，就需要开启新的会话，但新会话不会自动继承之前的内容。

长时间运行 Agent 的核心挑战在于：它们必须在离散的会话中工作，而每个新会话开始时对之前发生的事情没有任何记忆。想象一个由轮班工程师组成的软件项目，每个新来的工程师对上一班发生了什么毫无记忆。由于 Context Window 是有限的，而大多数复杂项目无法在单个窗口内完成，Agent 需要一种方法来弥合编码会话之间的鸿沟。

我们开发了一个双重解决方案，使 Claude Agent SDK 能够在多个 Context Window 之间有效工作：

  1. Initializer Agent（初始化 Agent）：在首次运行时设置环境

  2. Coding Agent（编码 Agent）：在每个会话中逐步推进，并为下一个会话留下清晰的记录

---

### 长时间运行 Agent 的问题

Claude Agent SDK 是一个强大的通用 Agent Harness（控制框架），擅长编码以及其他需要模型使用工具来收集上下文、规划和执行的任务。它具有 Compaction（上下文压缩）等上下文管理功能，使 Agent 能够在不耗尽 Context Window 的情况下持续工作。

> 💡 术语解释 - Compaction（上下文压缩）:

> 当对话接近 Context Window 上限时，系统会自动将之前的对话内容压缩成摘要，从而腾出空间继续工作。类似于你读完一本书后写了一份读书笔记，然后只带着笔记继续工作。

然而，Compaction 并不够用。开箱即用地使用 Claude Agent SDK，即使是最前沿的编码模型（如 Opus 4.5），在多个 Context Window 上循环运行时，如果只给一个高级提示词（如「构建一个 claude.ai 的克隆」），也无法构建出生产级质量的 Web 应用。

Claude 的失败表现为两种模式：

#### 失败模式一：试图一次性完成所有事

Agent 倾向于一次做太多事情——本质上是试图「一枪搞定」整个应用。这通常导致模型在实现过程中耗尽上下文，留给下一个会话的是一个功能实现了一半且没有文档记录的状态。Agent 只能猜测之前发生了什么，并花费大量时间试图让基本应用重新运行起来。

> 💡 举例说明:

> 就像让你一天之内建一栋房子，你开始砌墙、装窗户、布电线……但到天黑时墙只砌了一半，电线也没接完。第二天来的人不知道你做到了哪一步，只能花半天时间搞清楚状况，而不是继续建设。

#### 失败模式二：过早宣布完成

在项目后期，另一个 Agent 实例会环顾四周，看到已经取得了一些进展，就宣布工作完成了——即使很多功能还未实现。

---

### 解决方案：双重 Agent 架构

我们将问题分解为两部分：

  1. 设置初始环境：为所有功能需求奠定基础，引导 Agent 逐步、逐功能地工作

  2. 引导增量推进：让每个 Agent 在每个会话中取得增量进展，并在会话结束时保持环境的干净状态（Clean State）

> 💡 术语解释 - Clean State（干净状态）:

> 指代码处于适合合并到主分支的状态：没有重大 bug，代码有序且文档齐全。就像你下班前把工位收拾整齐，第二天来的人可以直接开始新工作，而不用先清理你的烂摊子。

具体实现：

  1. Initializer Agent：第一次 Agent 会话使用专门的提示词，要求模型设置初始环境：一个 init.sh 脚本、一个 claude-progress.txt 进度日志文件，以及一个显示添加了哪些文件的初始 git commit

  2. Coding Agent：后续每个会话要求模型取得增量进展，然后留下结构化的更新记录

> 🔑 核心洞察: 关键在于找到一种方法，让 Agent 在启动新的 Context Window 时能快速理解工作状态。这是通过 claude-progress.txt 文件配合 git 历史来实现的。这些实践灵感来源于高效软件工程师的日常工作方式。

### 环境管理详解

#### 功能清单（Feature List）

为了解决 Agent 一次性搞定或过早认为项目完成的问题，我们让 Initializer Agent 编写一份全面的功能需求文件，对用户的初始提示词进行扩展。在 claude.ai 克隆的例子中，这意味着超过 200 个功能，例如「用户可以打开新聊天、输入查询、按回车、看到 AI 回复」。

这些功能最初全部标记为「失败」（passes: false），以便后续的 Coding Agent 有一个清晰的功能完整性概览。

```json
{
  "category": "functional",
  "description": "新建聊天按钮创建一个新对话",
  "steps": [
    "导航到主界面",
    "点击'新建聊天'按钮",
    "验证创建了新对话",
    "检查聊天区域是否显示欢迎状态",
    "验证对话是否出现在侧边栏"
  ],
  "passes": false
}
```

我们要求 Coding Agent 只能通过更改 passes 字段的状态来编辑此文件，并使用措辞强烈的指令，如「删除或编辑测试是不可接受的，因为这可能导致功能缺失或有 bug」。经过实验，我们选择使用 JSON 格式，因为模型不太可能不当更改或覆盖 JSON 文件（相比 Markdown 文件）。

#### 增量推进（Incremental Progress）

在搭建好初始环境后，Coding Agent 的下一次迭代被要求每次只处理一个功能。这种增量方法被证明是解决 Agent 倾向于一次做太多事情的关键。

在实现增量推进后，模型在代码更改后保持环境干净状态仍然至关重要。在实验中，我们发现引发这种行为的最佳方式是要求模型将进展提交到 git（附带描述性 commit message），并在进度文件中写入进展摘要。这使得模型可以使用 git 来回退错误的代码更改并恢复代码库的工作状态。

> 💡 举例说明:

> 就像写论文时，每完成一个小节就保存一个版本（git commit），并写一句「完成了第三章第一节」。如果后面发现方向错了，可以随时回退到之前的版本，而不是从头再来。

#### 测试（Testing）

我们观察到的另一个主要失败模式是 Claude 倾向于在没有适当测试的情况下将功能标记为完成。在没有明确提示的情况下，Claude 会进行代码更改，甚至使用单元测试或 curl 命令对开发服务器进行测试，但无法识别功能是否端到端地正常工作。

在构建 Web 应用的场景中，一旦明确提示 Claude 使用 Browser Automation（浏览器自动化）工具并像人类用户一样进行所有测试，它在端到端验证功能方面就做得很好。

> 💡 术语解释 - Browser Automation（浏览器自动化）:

> 使用程序（如 Puppeteer）自动控制浏览器进行操作，模拟真实用户的点击、输入、滚动等行为。Claude 通过 Puppeteer MCP Server 来实现这一点。

为 Claude 提供这些测试工具显著提高了性能，因为 Agent 能够识别并修复仅从代码中看不明显的 bug。

不过仍有一些局限性，比如 Claude 的视觉能力和浏览器自动化工具的限制使其难以识别每种 bug。例如，Claude 无法通过 Puppeteer MCP 看到浏览器原生的 alert 弹窗，依赖这些弹窗的功能因此更容易出 bug。

### 快速上手（Getting Up to Speed）

有了以上所有设置，每个 Coding Agent 被要求执行一系列步骤来了解当前状况：

  1. 运行 pwd 查看你正在工作的目录

  2. 阅读 git 日志和进度文件，了解最近的工作进展

  3. 阅读功能清单文件，选择最高优先级且尚未完成的功能开始工作

这种方法在每个会话中为 Claude 节省了一些 token，因为它不需要弄清楚如何测试代码。同时，让 Initializer Agent 编写一个可以运行开发服务器的 init.sh 脚本也很有帮助。

在 claude.ai 克隆的例子中，Agent 总是先启动本地开发服务器，并使用 Puppeteer MCP 开始新聊天、发送消息并接收回复。这确保了 Claude 能快速识别应用是否处于损坏状态，并立即修复任何现有 bug。

一个典型的会话开始时的流程如下：

```javascript
[Assistant] 我先了解一下当前项目的状态。
[Tool Use] <bash - pwd>
[Tool Use] <read - claude-progress.txt>
[Tool Use] <read - feature_list.json>
[Assistant] 让我查看 git 日志了解最近的工作。
[Tool Use] <bash - git log --oneline -20>
[Assistant] 检查是否有 init.sh 脚本来重启服务器。
<启动开发服务器>
[Assistant] 很好！现在让我导航到应用并验证基本功能是否正常。
<测试基本功能>
[Assistant] 基于验证测试，基本功能运行良好。
核心聊天功能、主题切换、对话加载和错误处理都正常工作。
现在让我全面查看 tests.json 文件，了解接下来需要实现什么。
<开始处理新功能>
```

### 未来工作

这项研究展示了长时间运行 Agent Harness 中的一组可能解决方案，使模型能够在多个 Context Window 之间取得增量进展。但仍有一些开放性问题：

  1. 单 Agent vs 多 Agent 架构：目前尚不清楚是单个通用编码 Agent 在所有场景下表现最好，还是通过多 Agent 架构（如专门的测试 Agent、质量保证 Agent、代码清理 Agent）能获得更好的性能

  2. 领域泛化：目前的演示针对全栈 Web 应用开发进行了优化。未来的方向是将这些发现推广到其他领域，如科学研究或金融建模

---

### 关键要点

  1. 不要让 Agent 一次性做太多事 —— 增量推进是关键

  2. 使用结构化文件（JSON > Markdown） 来跟踪功能状态

  3. 每个会话结束时保持代码干净 —— 就像优秀的工程师每天下班前做的那样

  4. 使用 git 作为 Agent 的「记忆」 —— commit message + 进度文件

  5. 端到端测试不可或缺 —— 仅靠单元测试不够，需要像真实用户一样测试

  6. Initializer Agent 和 Coding Agent 分工明确 —— 一个搭台，一个唱戏

---

> 📝 译者注: 本文的核心思想是借鉴人类工程师的最佳实践（轮班交接、进度文档、版本控制、逐步推进）来解决 AI Agent 在长时间运行任务中的记忆和连续性问题。这些原则不仅适用于 Claude Agent SDK，也适用于任何需要跨会话保持状态的 AI Agent 系统。
