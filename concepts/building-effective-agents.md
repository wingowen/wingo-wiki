## 构建高效 Agent

> 原文: Building effective agents

> 作者: Erik Schluntz, Barry Zhang（Anthropic）

> 发布日期: 2024-12-19

> 翻译日期: 2026-04-14

---

### 概述

过去一年，我们与数十个跨行业的团队合作构建 LLM Agent。我们发现，最成功的实现并非使用复杂的框架或专用库，而是使用简单、可组合的模式。

> 💡 术语解释 - Agent（智能体）:

> 在本文语境中，Agent 指的是能够自主使用工具、规划步骤并执行任务以达成目标的 AI 系统。与简单的"一问一答"不同，Agent 可以进行多轮交互和复杂推理。

---

### 什么是 Agent？

"Agent" 有多种定义方式。一些客户将 Agent 定义为在较长时间内独立运行、使用各种工具完成复杂任务的完全自主系统。另一些则用这个词来描述遵循预定义工作流的更具规定性的实现。在 Anthropic，我们将所有这些变体归类为 Agentic Systems（智能体系统），但在架构上做出重要区分：

  - Workflows（工作流）：通过预定义的代码路径编排 LLM 和工具的系统

  - Agents（智能体）：LLM 动态地指导自身流程和工具使用，自主控制如何完成任务

> 💡 举例说明 - Workflow vs Agent:

> - Workflow 就像工厂流水线：每个工人（LLM 调用）按固定顺序完成自己的工序，整体流程是预先设计好的

> - Agent 就像一个独立的项目经理：根据任务需要自主决定下一步做什么、用什么工具、找谁帮忙

---

### 何时使用（以及何时不使用）Agent

使用 LLM 构建应用时，我们建议寻找最简单的解决方案，只在需要时才增加复杂性。这可能意味着根本不构建 Agentic System。Agentic System 通常以延迟和成本换取更好的任务性能，你需要考虑这种权衡何时合理。

当确实需要更多复杂性时，Workflows 为定义明确的任务提供可预测性和一致性，而 Agents 则在需要灵活性和模型驱动的决策时是更好的选择。然而对于许多应用，通过检索和上下文示例优化单次 LLM 调用通常就足够了。

---

### 何时以及如何使用框架

目前有许多框架可以简化 Agentic System 的实现，包括：

  - Claude Agent SDK

  - Strands Agents SDK by AWS

  - Rivet（拖拽式 GUI LLM 工作流构建器）

  - Vellum（构建和测试复杂工作流的 GUI 工具）

这些框架通过简化调用 LLM、定义和解析工具、链式调用等标准底层任务，降低了入门门槛。然而，它们往往创建了额外的抽象层，可能掩盖底层的提示词和响应，使调试更加困难。它们还可能诱使你在简单设置就足够的情况下添加不必要的复杂性。

> 🔑 建议: 开发者应首先直接使用 LLM API——许多模式只需几行代码即可实现。如果使用框架，务必理解底层代码。对底层实现的不正确假设是客户错误的常见来源。

### 构建模块、工作流和 Agent

#### 构建模块：增强型 LLM（Augmented LLM）

Agentic System 的基本构建模块是增强了检索、工具和记忆等能力的 LLM。我们当前的模型可以主动使用这些能力——生成自己的搜索查询、选择适当的工具、决定保留哪些信息。

我们建议在实现中关注两个关键方面：将这些能力量身定制到你的特定用例，并确保它们为 LLM 提供简单、文档完善的接口。

> 💡 术语解释 - Augmented LLM（增强型 LLM）:

> 就像给一个聪明的员工配备了电脑、搜索引擎和文件柜。LLLM 本身很聪明，但有了这些工具后，它可以获取外部信息、执行操作、记住之前的工作，能力大幅提升。

#### 工作流：Prompt Chaining（提示链）

Prompt Chaining 将任务分解为一系列步骤，每个 LLM 调用处理前一个的输出。你可以在任何中间步骤添加程序化检查（下图中的"gate"），以确保流程仍在正轨上。

适用场景：任务可以轻松、干净地分解为固定子任务时。主要目标是通过让每个 LLM 调用处理更简单的任务，以延迟换取更高的准确性。

示例：

  - 生成营销文案，然后将其翻译成不同语言

  - 编写文档大纲 → 检查大纲是否符合标准 → 基于大纲撰写文档

#### 工作流：Routing（路由分发）

Routing 对输入进行分类，并将其引导到专门的后续任务中。这种工作流实现了关注点分离，可以构建更专门的提示词。

适用场景：复杂任务中有明显不同的类别，且分类可以由 LLM 或传统分类模型准确处理时。

示例：

  - 将不同类型的客户服务查询（一般问题、退款请求、技术支持）引导到不同的下游流程

  - 将简单/常见问题路由到更小、更经济的模型（如 Claude Haiku 4.5），将困难/罕见问题路由到更强大的模型（如 Claude Sonnet 4.5）

#### 工作流：Parallelization（并行化）

LLM 有时可以同时处理一个任务，并以程序化方式汇总其输出。并行化有两种关键变体：

  - Sectioning（分段）：将任务分解为独立子任务并行运行

  - Voting（投票）：多次运行同一任务以获得多样化输出

适用场景：子任务可以并行化以提速，或需要多个视角/尝试以获得更高置信度的结果时。

示例：

  - Sectioning：实现护栏——一个模型实例处理用户查询，另一个审查不当内容

  - Voting：审查代码漏洞——多个不同提示词审查同一段代码，发现问题则标记

#### 工作流：Orchestrator-Workers（编排者-工作者模式）

中央 LLM 动态分解任务，将其委派给工作者 LLM，并综合它们的结果。

> 💡 举例说明:

> 就像一个项目经理把一个大项目拆分成多个小任务，分配给不同的团队成员，最后汇总所有人的成果。与并行化的关键区别在于：子任务不是预定义的，而是由编排者根据具体输入动态决定的。

适用场景：无法预测所需子任务的复杂任务（例如编码中需要修改的文件数量和性质取决于具体任务）。

#### 工作流：Evaluator-Optimizer（评估器-优化器模式）

一个 LLM 调用生成响应，另一个在循环中提供评估和反馈。

适用场景：有明确评估标准，且迭代改进能带来可衡量价值时。

示例：

  - 文学翻译——翻译 LLM 可能最初无法捕捉某些细微差别，但评估 LLM 可以提供有用的批评

  - 需要多轮搜索和分析的复杂搜索任务

#### Agent（自主智能体）

Agent 开始工作时，要么接收人类用户的命令，要么与人类进行交互讨论。任务明确后，Agent 自主规划和执行，可能返回向人类获取更多信息或判断。执行过程中，Agent 必须在每一步从环境中获取"真实信息"（如工具调用结果或代码执行结果）来评估进展。

> ⚠️ 注意: Agent 的自主性意味着更高的成本和复合错误的潜在风险。我们建议在沙箱环境中进行广泛测试，并配合适当的护栏。

---

### 组合和定制这些模式

这些构建模块不是规定性的。它们是开发者可以根据不同用例进行塑造和组合的常见模式。成功的关键——与任何 LLM 功能一样——是衡量性能并迭代实现。

> 🔑 核心原则: 只有在复杂性被证明能改善结果时，才应增加复杂性。

---

### 总结

在 LLM 领域，成功不在于构建最复杂的系统，而在于为你的需求构建正确的系统。从简单的提示词开始，通过全面评估进行优化，只有在更简单的解决方案不够用时才添加多步骤的 Agentic System。

实现 Agent 时，我们遵循三个核心原则：

  1. 保持 Agent 设计的简洁性

  2. 通过明确展示 Agent 的规划步骤来优先考虑透明性

  3. 通过彻底的工具文档和测试来精心设计 Agent-计算机接口（ACI）
