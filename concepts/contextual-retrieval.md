## 介绍上下文检索 | Introducing Contextual Retrieval

> 发布日期：2024-09-19

AI 模型要在特定场景中发挥作用，通常需要获取背景知识。例如，客户支持聊天机器人需要了解其所服务的特定业务，而法律分析机器人则需要掌握大量过往案例。

开发者通常使用 RAG（Retrieval-Augmented Generation，检索增强生成）来增强 AI 模型的知识。RAG 是一种从知识库中检索相关信息并将其附加到用户提示词中的方法，能够显著提升模型的回答质量。问题在于，传统的 RAG 方案在编码信息时会丢失上下文，导致系统经常无法从知识库中检索到相关信息。

💡 术语解释：RAG（检索增强生成） — 一种将信息检索与大语言模型生成相结合的技术框架。系统先从外部知识库中检索与用户问题相关的文档片段，然后将这些片段作为上下文提供给模型，从而生成更准确、更有依据的回答。

在本文中，我们将介绍一种大幅改善 RAG 中检索步骤的方法。该方法名为「上下文检索（Contextual Retrieval）」，使用了两个子技术：上下文嵌入（Contextual Embeddings）和上下文 BM25（Contextual BM25）。该方法可将检索失败率降低 49%，结合 Reranking（重排序）后可降低 67%。这些检索准确率的显著提升，将直接转化为下游任务性能的改善。

你可以使用 Claude 轻松部署自己的上下文检索方案，请参考我们的 cookbook。

### 关于直接使用更长提示词的说明

有时最简单的方案就是最好的。如果你的知识库小于 200,000 个 token（约 500 页材料），你可以直接将整个知识库放入提供给模型的提示词中，无需使用 RAG 或类似方法。

几周前，我们为 Claude 发布了 Prompt Caching（提示词缓存），使这种方法的速度显著提升、成本大幅降低。开发者现在可以在 API 调用之间缓存常用提示词，延迟降低超过 2 倍，成本降低高达 90%（你可以通过阅读我们的 prompt caching cookbook 了解其工作原理）。

💡 术语解释：Prompt Caching（提示词缓存） — 一种优化技术，允许开发者缓存提示词中不变的部分（如系统指令、知识库内容），在后续 API 调用时直接复用缓存，从而减少重复处理的开销，降低延迟和成本。

然而，随着知识库的增长，你需要一个更具扩展性的方案。这就是上下文检索发挥作用的地方。

### RAG 基础：扩展到更大的知识库

对于无法放入上下文窗口的较大知识库，RAG 是典型的解决方案。RAG 通过以下步骤对知识库进行预处理：

  1. 将知识库（文档的「语料库」）分解为较小的文本块（chunks），通常不超过几百个 token；

  2. 使用 Embedding 模型（嵌入模型）将这些文本块转换为编码语义信息的向量嵌入（vector embeddings）；

  3. 将这些嵌入存储在向量数据库中，支持通过语义相似度进行搜索。

💡 术语解释：Embedding（嵌入） — 将文本转换为高维数值向量的过程。语义相近的文本在向量空间中距离更近，这使得计算机能够通过计算向量之间的距离来判断文本的相关性。

在运行时，当用户向模型输入查询时，向量数据库会根据与查询的语义相似度找到最相关的文本块。然后，这些最相关的文本块会被添加到发送给生成模型的提示词中。

虽然嵌入模型擅长捕捉语义关系，但它们可能会遗漏关键的精确匹配。幸运的是，有一种更早的技术可以在这些场景中提供帮助。BM25（Best Matching 25，最佳匹配 25）是一种使用词法匹配来查找精确词汇或短语匹配的排序函数。它对于包含唯一标识符或技术术语的查询特别有效。

💡 术语解释：BM25 — 一种基于词频的排序算法，是信息检索领域最经典的算法之一。它通过计算查询词在文档中出现的频率（TF）和在整个文档集合中的稀有程度（IDF）来评估文档与查询的相关性，擅长处理精确关键词匹配。

BM25 的工作原理建立在 TF-IDF（Term Frequency-Inverse Document Frequency，词频-逆文档频率）概念之上。TF-IDF 衡量一个词对文档集合中某个文档的重要程度。BM25 通过考虑文档长度并对词频应用饱和函数来改进这一方法，有助于防止常见词主导结果。

💡 举例说明：假设用户在技术支持数据库中查询「错误代码 TS-999」。嵌入模型可能会找到关于错误代码的通用内容，但可能遗漏精确的「TS-999」匹配。BM25 则会查找这个特定的文本字符串来识别相关文档。

RAG 方案可以通过结合嵌入和 BM25 技术来更准确地检索最适用的文本块，具体步骤如下：

  1. 将知识库（文档的「语料库」）分解为较小的文本块，通常不超过几百个 token；

  2. 为这些文本块创建 TF-IDF 编码和语义嵌入；

  3. 使用 BM25 基于精确匹配找到排名靠前的文本块；

  4. 使用嵌入基于语义相似度找到排名靠前的文本块；

  5. 使用 Rank Fusion（排序融合）技术合并并去重第（3）和第（4）步的结果；

  6. 将排名前 K 的文本块添加到提示词中以生成回答。

一个标准的使用嵌入和 BM25 进行信息检索的 RAG 系统。TF-IDF（词频-逆文档频率）衡量词语重要性，是 BM25 的基础。

通过利用 BM25 和嵌入模型，传统 RAG 系统能够提供更全面、更准确的结果，在精确词项匹配与更广泛的语义理解之间取得平衡。

这种方法使你能够以经济高效的方式扩展到海量知识库，远远超出单个提示词所能容纳的范围。但这些传统 RAG 系统存在一个重大局限：它们经常破坏上下文。

### 传统 RAG 中的上下文困境

在传统 RAG 中，文档通常被分割为较小的文本块以便高效检索。虽然这种方法在许多应用中效果良好，但当单个文本块缺乏足够的上下文时，就会引发问题。

💡 举例说明：假设你的知识库中嵌入了金融信息（例如美国 SEC 文件），你收到以下问题：「ACME Corp 在 2023 年第二季度的收入增长是多少？」

一个相关的文本块可能包含这样的文本：「该公司的收入较上一季度增长了 3%。」然而，这个文本块本身并未指明它涉及哪家公司，也没有说明相关时间段，这使得检索正确信息或有效利用信息变得困难。

### 介绍上下文检索

上下文检索通过在嵌入前为每个文本块添加特定于该文本块的解释性上下文（「上下文嵌入」）以及创建 BM25 索引（「上下文 BM25」）来解决这一问题。

让我们回到 SEC 文件集合的例子。以下是文本块如何被转换的示例：

```python
original_chunk = "The company's revenue grew by 3% over the previous quarter."

contextualized_chunk = "This chunk is from an SEC filing on ACME corp's performance in Q2 2023; the previous quarter's revenue was $314 million. The company's revenue grew by 3% over the previous quarter."
```

值得注意的是，过去已经有人提出过使用上下文来改善检索的方法。其他提案包括：为文本块添加通用文档摘要（我们进行了实验，发现效果非常有限）、假设文档嵌入以及基于摘要的索引（我们进行了评估，发现性能较低）。这些方法与本文提出的方法有所不同。

### 实现上下文检索

当然，手动为知识库中成千上万甚至数百万个文本块添加注释是不现实的。为了实现上下文检索，我们借助了 Claude。我们编写了一个提示词，指示模型利用整篇文档的上下文，为每个文本块提供简洁的、特定于该文本块的上下文说明。我们使用以下 Claude 3 Haiku 提示词来为每个文本块生成上下文：

```javascript
<document>
{{WHOLE_DOCUMENT}}
</document>
Here is the chunk we want to situate within the whole document
<chunk>
{{CHUNK_CONTENT}}
</chunk>
Please give a short succinct context to situate this chunk within the overall document for the purposes of improving search retrieval of the chunk. Answer only with the succinct context and nothing else.
```

生成的上下文文本通常为 50-100 个 token，会在嵌入文本块和创建 BM25 索引之前添加到文本块前面。

### 使用 Prompt Caching 降低上下文检索的成本

得益于我们上文提到的 Prompt Caching 特性，使用 Claude 可以以极低的成本实现上下文检索。有了提示词缓存，你不需要为每个文本块都传入参考文档。你只需将文档加载到缓存中一次，然后引用之前缓存的内容即可。假设每个文本块 800 个 token、每篇文档 8k 个 token、上下文指令 50 个 token、每个文本块的上下文 100 个 token，生成上下文化文本块的一次性成本为每百万文档 token $1.02。

### 实现注意事项

在实现上下文检索时，有几点需要考虑：

  - 分块边界（Chunk Boundaries）：考虑如何将文档分割为文本块。分块大小、分块边界和分块重叠的选择都会影响检索性能^1。

  - 嵌入模型：虽然上下文检索在我们测试的所有嵌入模型上都提升了性能，但某些模型可能受益更多。我们发现 Gemini 和 Voyage 的嵌入效果特别好。

  - 自定义上下文化提示词：虽然我们提供的通用提示词效果良好，但你可能会通过针对特定领域或用例定制的提示词获得更好的效果（例如，包含一个术语表，列出可能仅在知识库其他文档中定义的关键术语）。

  - 文本块数量：向上下文窗口中添加更多文本块会增加包含相关信息的概率。然而，更多信息可能会分散模型的注意力，因此存在一个上限。我们尝试了传递 5、10 和 20 个文本块，发现使用 20 个是这些选项中效果最好的（参见附录中的对比），但值得在你的用例上进行实验。

  - 始终运行评估：通过向模型传递上下文化后的文本块并区分上下文和原始文本块内容，可能会进一步改善回答生成效果。

### 使用 Reranking 进一步提升性能

作为最后一步，我们可以将上下文检索与另一种技术结合，以获得更多的性能提升。在传统 RAG 中，AI 系统搜索其知识库以找到潜在相关的信息文本块。面对大型知识库，初始检索通常会返回大量文本块——有时多达数百个——其相关性和重要性各不相同。

💡 术语解释：Reranking（重排序） — 一种后处理技术，在初始检索返回一批候选文本块后，使用专门的排序模型对它们进行重新评分和排序，确保只有最相关的文本块被传递给生成模型，从而提高回答质量并降低成本。

Reranking 是一种常用的过滤技术，用于确保只有最相关的文本块被传递给模型。Reranking 提供了更好的回答，同时降低了成本和延迟，因为模型处理的信息更少。关键步骤如下：

  1. 执行初始检索以获取排名靠前的潜在相关文本块（我们使用了排名前 150）；

  2. 将排名前 N 的文本块连同用户查询一起传入 Reranking 模型；

  3. 使用 Reranking 模型根据每个文本块与提示词的相关性和重要性进行评分，然后选择排名前 K 的文本块（我们使用了排名前 20）；

  4. 将排名前 K 的文本块作为上下文传入模型以生成最终结果。

结合上下文检索和 Reranking 以最大化检索准确率。

### 性能提升

市场上有多种 Reranking 模型。我们使用 Cohere reranker 进行了测试。Voyage 也提供了 reranker，但我们没有时间进行测试。我们的实验表明，在各个领域中，添加 Reranking 步骤都能进一步优化检索效果。

具体而言，我们发现经过 Reranking 的上下文嵌入和上下文 BM25 将排名前 20 的文本块检索失败率降低了 67%（5.7% -> 1.9%）。

经过 Reranking 的上下文嵌入和上下文 BM25 将排名前 20 的文本块检索失败率降低了 67%。

### 成本和延迟考量

Reranking 的一个重要考量是对延迟和成本的影响，尤其是在对大量文本块进行重排序时。由于 Reranking 在运行时增加了一个额外步骤，即使 reranker 并行对所有文本块进行评分，也必然会增加少量延迟。在重排序更多文本块以获得更好性能与重排序较少文本块以降低延迟和成本之间，存在一个固有的权衡。我们建议在你的特定用例上尝试不同的设置，以找到合适的平衡点。

### 结论

我们进行了大量测试，比较了上述所有技术的不同组合（嵌入模型、是否使用 BM25、是否使用上下文检索、是否使用 Reranker 以及检索的前 K 个结果总数），覆盖了各种不同的数据集类型。以下是我们发现的总结：

  - 嵌入 + BM25 优于单独使用嵌入；

  - Voyage 和 Gemini 在我们测试的嵌入中表现最佳；

  - 向模型传递排名前 20 的文本块比仅传递前 10 或前 5 更有效；

  - 为文本块添加上下文能大幅提升检索准确率；

  - 使用 Reranking 优于不使用 Reranking；

  - 所有这些优势可以叠加：为了最大化性能提升，我们可以结合上下文嵌入（来自 Voyage 或 Gemini）、上下文 BM25，加上 Reranking 步骤，并将 20 个文本块添加到提示词中。
