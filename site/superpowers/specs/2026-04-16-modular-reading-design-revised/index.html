    长文章模块化阅读设计方案（修订版）

    1. 项目背景

    当前 raw/articles 目录下存在一些较长的文章，影响阅读体验和维护效率。通过分析发现，以下文章超过了 10,000 字符：

    • mcp-deep-dive.md：19,114 字符
    • agent-framework-practice.md：15,021 字符
    • swe-bench.md：13,938 字符

    这些长文章需要通过模块化拆分和双链连接来提高可读性和可维护性，同时确保不影响现有的 llm wiki 维护结构。

    2. 现有系统分析

    2.1 项目结构

    • raw/articles/：原始文章目录（从外部导入的原始内容）
    • concepts/：概念页面目录（处理后的 wiki 页面）
    • content/：内容页面目录（Hugo 站点内容）
    • entities/：实体页面目录
    • comparisons/：比较页面目录
    • queries/：查询页面目录

    2.2 维护工作流

    • 内容来源：从 Notion 导入原始内容到 raw/articles/
    • 处理流程：通过 importer.py 工具处理原始内容，生成 wiki 页面
    • 发布流程：使用 Hugo 生成静态网站
    • 索引管理：通过 index.md 维护页面索引
    • 日志管理：通过 log.md 记录维护操作

    2.3 双链机制

    • 使用 [[wikilinks]] 格式创建页面间链接
    • 有 layouts/shortcodes/wikilink.html 处理双链渲染
    • 支持 [[link|display text]] 格式的链接

    2.4 现有规范

    • 文件命名：小写，连字符分隔，无空格
    • 页面结构：YAML frontmatter + 内容 + 相关链接
    • 页面拆分阈值：超过 ~200 行的页面应拆分为子主题

    3. 模块化方案

    3.1 方案 1：基于章节的模块化（推荐）

    描述：将长文章按章节拆分为多个文件，主文件包含目录和概述，所有文件保持在 raw/articles/ 目录下。

    优点： - 结构清晰，符合原始文章的组织结构 - 导航方便，可通过双链快速跳转 - 维护成本低，修改特定章节只需更新对应文件 - 不影响现有维护工作流，保持与 importer.py 的兼容性

    缺点： - 文件数量增加 - 需要保持章节间的一致性

    实施步骤： 1. 分析文章结构，识别主要章节 2. 保留原主文件，添加目录和概述 3. 为每个章节创建单独的文件 4. 使用双链连接主文件和章节文件 5. 在章节文件中添加返回主文件的链接

    3.2 方案 2：使用子目录的模块化

    描述：为长文章创建子目录，将章节文件放在子目录中。

    优点： - 文件组织更清晰，避免主目录文件过多 - 可以更好地管理相关资源

    缺点： - 可能影响现有的文件结构和导入流程 - 增加了目录管理的复杂度

    实施步骤： 1. 为长文章创建子目录 2. 将主文件和章节文件放在子目录中 3. 更新双链路径以适应新的目录结构

    3.3 方案 3：仅在渲染层模块化

    描述：保持原始文件不变，仅在渲染时通过 Hugo 短代码实现模块化导航。

    优点： - 不改变现有文件结构 - 维护成本最低

    缺点： - 原始文件仍然很长，不利于编辑和维护 - 依赖 Hugo 渲染能力，本地阅读体验不佳

    实施步骤： 1. 在长文章中添加目录标记 2. 创建 Hugo 短代码实现章节导航 3. 在渲染时生成模块化导航

    4. 推荐方案

    推荐方案：方案 1 - 基于章节的模块化

    推荐理由： - 保持兼容性：所有文件保持在 raw/articles/ 目录下，不影响现有的 importer.py 工作流 - 结构清晰：符合原始文章的组织结构，便于理解和维护 - 实施简单：不需要修改现有工具和流程，只需添加新文件 - 导航方便：使用双链可以快速在不同模块之间跳转 - 符合规范：符合 SCHEMA.md 中的页面拆分原则

    放弃原因： - 方案 2 可能影响现有的文件结构和导入流程，增加了不必要的复杂度 - 方案 3 虽然维护成本最低，但没有解决原始文件过长的问题，不利于编辑和维护

    5. 具体实施细节

    5.1 文件命名规范

    • 主文件：保持原文件名，如 agent-framework-practice.md
    • 章节文件：原文件名-章节slug.md，如 agent-framework-practice-architecture.md
    • 章节 slug：使用小写，连字符分隔，去除特殊字符

    5.2 目录结构

├── agent-framework-practice.md         # 主文件
├── agent-framework-practice-architecture.md   # 章节文件
├── agent-framework-practice-elements.md       # 章节文件

    5.5 双链使用规范

    • 主文件：使用双链链接到所有章节
    • 章节文件：使用双链链接回主文件和相关章节
    • 保持一致性：确保双链的准确性和一致性
    • 遵循现有规范：使用与现有页面相同的双链格式

    6. 实施计划

    6.1 优先级

    1. mcp-deep-dive.md（19,114 字符）
    2. agent-framework-practice.md（15,021 字符）
    3. swe-bench.md（13,938 字符）

    6.2 步骤

    1. 分析文章结构：使用 modularize.py 脚本分析文章结构和章节
    2. 更新主文件：保留原主文件，添加目录和概述
    3. 创建章节文件：为每个章节创建单独的文件
    4. 添加双链：在主文件和章节文件中添加相应的双链
    5. 测试链接：确保所有双链都能正确工作
    6. 更新索引：将新创建的文件添加到 index.md
    7. 记录日志：在 log.md 中记录模块化操作

    7. 维护策略

    7.1 与现有工作流的集成

    • 保持兼容性：所有文件保持在 raw/articles/ 目录下，不影响 importer.py 的工作流
    • 同步更新：当原始文章更新时，同步更新主文件和相关章节文件
    • 版本控制：使用相同的 updated 日期标记相关文件

    7.2 质量控制

    • 定期检查：定期检查双链的有效性和一致性
    • 内容同步：确保章节文件的内容与主文件的目录一致
    • 命名规范：保持文件命名和结构的一致性

    7.3 文档规范

    • 遵循现有规范：遵循 SCHEMA.md 中的文档规范
    • 统一格式：保持 YAML frontmatter 的一致性
    • 标准双链：使用统一的双链格式和命名规范

    8. 预期效果

    通过模块化拆分和双链连接，预期达到以下效果：

    • 提高可读性：将长文章拆分为多个小模块，每个模块专注于一个主题
    • 便于导航：使用双链可以快速在不同模块之间跳转
    • 易于维护：修改特定内容时只需更新对应的模块文件
    • 知识连接：通过双链建立模块之间的关联，形成知识网络
    • 减少认知负担：读者可以根据需要选择性阅读特定模块
    • 保持兼容性：不影响现有的 llm wiki 维护结构和工作流

    9. 风险评估

    9.1 潜在风险

    • 链接失效：如果文件命名或路径发生变化，可能导致双链失效
    • 内容不一致：如果更新时没有同步所有相关文件，可能导致内容不一致
    • 文件数量增加：模块化会增加文件数量，可能影响项目的管理
    • 工作流影响：可能影响现有的 importer.py 导入流程

    9.2 风险缓解

    • 使用脚本：使用脚本自动生成和检查双链
    • 统一更新：建立统一的更新机制，确保相关文件同步更新
    • 保持结构：所有文件保持在 raw/articles/ 目录下，不改变现有结构
    • 测试验证：在实施后测试所有链接和导入流程

    10. 结论

    通过基于章节的模块化方案，可以有效提高长文章的可读性和可维护性，同时保持与现有 llm wiki 维护结构的兼容性。该方案符合现有项目的组织结构和规范，实施简单，易于理解和维护。通过双链连接，可以建立模块之间的关联，形成一个结构化的知识体系，为读者提供更好的阅读体验。

    该方案的关键优势在于： - 最小侵入性：不改变现有文件结构和工作流 - 高兼容性：与现有的 importer.py 工具和 Hugo 渲染流程完全兼容 - 易实施性：只需添加新文件，不需要修改现有工具和流程 - 可扩展性：可以根据需要扩展到其他长文章

    通过这种方式，可以在不影响现有维护结构的情况下，实现长文章的模块化阅读，提高整个 wiki 系统的质量和可用性。
