## AI Agent 的有效上下文工程 | Effective context engineering for AI agents

> 原文链接：https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

> 作者：Prithvi Rajasekaran, Ethan Dixon, Carly Ryan, Jeremy Hadfield（Anthropic Engineering）

> 发布日期：2025-09-29

> 翻译日期：2026-04-14

---

上下文（Context）是 AI Agent 的一项关键但有限的资源。在本文中，我们将探讨有效策展和管理驱动 Agent 运行的上下文的策略。

在 Prompt Engineering（提示词工程）成为应用 AI 关注焦点的几年之后，一个新术语逐渐崭露头角：Context Engineering（上下文工程）。使用语言模型构建应用，已经不再仅仅是找到提示词中合适的措辞，而更多地是在回答一个更广泛的问题——"什么样的上下文配置最有可能产生我们期望的模型行为？"

💡 术语解释：Context（上下文）指的是从大语言模型（LLM）采样时包含的所有 Token（词元）的集合。上下文工程的核心问题是在 LLM 的固有约束下优化这些 Token 的效用，以持续达成期望的结果。有效驾驭 LLM 通常需要"以上下文的方式思考"——也就是说，要考虑 LLM 在任何给定时刻可用的整体状态，以及该状态可能产生哪些潜在行为。

在本文中，我们将探索上下文工程这一新兴艺术，并为构建可操控、高效的 Agent 提供一个更精细的心智模型。

### 上下文工程 vs. 提示词工程

在 Anthropic，我们将上下文工程视为提示词工程的自然演进。提示词工程指的是编写和组织 LLM 指令以获得最佳结果的方法（参见我们的文档了解概述和实用的提示词工程策略）。上下文工程指的是在 LLM 推理过程中策展和维护最优 Token（信息）集合的策略，包括提示词之外可能进入上下文的所有其他信息。

在 LLM 工程的早期，编写提示词是 AI 工程工作中最大的组成部分，因为日常聊天交互之外的大多数用例都需要为一次性分类或文本生成任务优化提示词。顾名思义，提示词工程的主要焦点是如何编写有效的提示词，尤其是系统提示词。然而，随着我们向构建在多轮推理和更长的时间跨度上运行的更强大的 Agent 迈进，我们需要管理整个上下文状态的策略（系统指令、工具、Model Context Protocol (MCP)、外部数据、消息历史等）。

在循环中运行的 Agent 会生成越来越多可能与下一轮推理相关的数据，而这些信息必须被循环地精炼。上下文工程是从不断演变的可能信息宇宙中，策展将哪些内容放入有限的上下文窗口的艺术与科学。

与编写提示词这一离散任务不同，上下文工程是迭代的，策展阶段发生在每次我们决定传递什么给模型的时候。

### 为什么上下文工程对构建强大的 Agent 至关重要

尽管 LLM 速度越来越快，能够处理越来越大的数据量，但我们观察到，LLM 和人类一样，在某个临界点会失去焦点或产生困惑。对"大海捞针"式基准测试的研究揭示了 Context Rot（上下文衰减）的概念：随着上下文窗口中 Token 数量的增加，模型从该上下文中准确回忆信息的能力会下降。

💡 术语解释：Context Rot（上下文衰减）指的是当上下文窗口中的 Token 数量增长时，模型准确回忆上下文中信息的能力逐渐下降的现象。这类似于人类在处理过多信息时注意力分散的情况。

虽然一些模型的退化比其他模型更温和，但这一特征在所有模型中都存在。因此，上下文必须被视为一种边际收益递减的有限资源。就像人类拥有有限的工作记忆容量一样，LLM 在解析大量上下文时也有一个"注意力预算"（Attention Budget）。引入的每一个新 Token 都会消耗一部分预算，这增加了仔细策展 LLM 可用 Token 的必要性。

这种注意力稀缺性源于 LLM 的架构约束。LLM 基于 Transformer 架构，该架构使每个 Token 都能关注到上下文中的每个其他 Token。对于 n 个 Token，这会产生 n² 的成对关系。

随着上下文长度的增加，模型捕获这些成对关系的能力被拉伸变薄，在上下文大小和注意力焦点之间产生了自然的张力。此外，模型从训练数据分布中发展出注意力模式，其中较短的序列通常比较长的序列更常见。这意味着模型对上下文范围依赖关系的经验和专用参数更少。

💡 术语解释：Position Encoding Interpolation（位置编码插值）是一种允许模型通过将模型适配到最初训练的较小上下文来处理更长序列的技术，尽管在 Token 位置理解上会有一定退化。这些因素创造的是一个性能梯度而非硬性断崖：模型在较长上下文中仍然高度胜任，但在信息检索和长程推理方面可能表现出与较短上下文相比的精度下降。

这些现实意味着，精心设计的上下文工程对于构建强大的 Agent 是必不可少的。

### 有效上下文的解剖

鉴于 LLM 受限于有限的注意力预算，好的上下文工程意味着找到尽可能最小的、高信号 Token 集合，以最大化实现某种期望结果的可能性。实践这一原则说起来容易做起来难，但在以下部分中，我们将概述这一指导原则在上下文不同组件中的实际含义。

系统提示词（System Prompts）应该极其清晰，使用简单直接的语言，在适合 Agent 的"正确高度"呈现想法。正确的高度是两种常见失败模式之间的 Goldilocks 区间（恰到好处的区间）。一个极端是，工程师在提示词中硬编码复杂、脆弱的逻辑以引发精确的 Agent 行为。这种方法造成了脆弱性，并随时间增加了维护复杂性。另一个极端是，工程师有时提供模糊的、高层级的指导，未能给 LLM 提供期望输出的具体信号，或者错误地假设了共享上下文。最优高度在两者之间取得平衡：足够具体以有效引导行为，又足够灵活以为模型提供强有力的启发式指导。

💡 举例说明：一个过于具体的提示词可能写"如果用户问 X，则返回 Y；如果用户问 Z，则返回 W..."——这种 if-else 硬编码方式非常脆弱。一个过于模糊的提示词可能写"请帮助用户"——这没有给出足够的信号。一个好的提示词则写"你是代码审查助手，专注于识别安全漏洞和性能问题，使用以下格式输出..."

我们建议将提示词组织成不同的部分（如 <background_information>、<instructions>、## Tool guidance、## Output description 等），并使用 XML 标签或 Markdown 标题来划分这些部分，尽管随着模型能力的提升，提示词的确切格式可能变得不那么重要。

无论你决定如何构建系统提示词，都应该追求完全概述期望行为所需的最小信息集。（注意，最小不一定意味着简短；你仍然需要预先给 Agent 足够的信息以确保它遵循期望的行为。）最好的做法是先用最好的模型测试一个最小提示词，看看它在你的任务上表现如何，然后根据初始测试中发现的失败模式添加清晰的指令和示例来提高性能。

工具（Tools）允许 Agent 与其环境交互并在工作时引入新的、额外的上下文。因为工具定义了 Agent 与其信息/操作空间之间的契约，所以工具促进效率是极其重要的——既通过返回 Token 高效的信息，也通过鼓励高效的 Agent 行为。

在为 AI Agent 编写工具——用 AI Agent一文中，我们讨论了构建 LLM 能很好理解且功能重叠最小的工具。类似于设计良好的代码库中的函数，工具应该是自包含的、对错误有鲁棒性的，并且在预期用途上极其清晰。输入参数同样应该是描述性的、无歧义的，并发挥模型的固有优势。

我们看到的最常见的失败模式之一是臃肿的工具集，它们覆盖了太多功能或导致了关于使用哪个工具的模糊决策点。如果人类工程师不能明确地说出在给定情况下应该使用哪个工具，就不能期望 AI Agent 做得更好。正如我们稍后将讨论的，为 Agent 策展一个最小可行工具集也可以在长交互中实现更可靠的上下文维护和修剪。

提供示例，也称为 Few-shot Prompting（少样本提示），是一个众所周知的最佳实践，我们继续强烈建议使用。然而，团队经常将一大堆边缘情况塞进提示词中，试图阐述 LLM 应该遵循的每一条可能的规则。我们不推荐这样做。相反，我们建议策展一组多样化的、规范的示例，有效地描绘 Agent 的期望行为。对于 LLM 来说，示例就是"价值千言的图片"。

💡 术语解释：Few-shot Prompting（少样本提示）是指在提示词中提供少量示例来引导模型理解任务格式和期望输出的技术。与之相对的是 Zero-shot（零样本，不提供示例）和 One-shot（单样本，提供一个示例）。

我们对上下文不同组件（系统提示词、工具、示例、消息历史等）的总体建议是：深思熟虑，保持上下文信息丰富但紧凑。现在让我们深入探讨在运行时动态检索上下文。

### 上下文检索与 Agent 搜索

在构建高效的 AI Agent一文中，我们强调了基于 LLM 的工作流和 Agent 之间的区别。自那篇文章以来，我们倾向于对 Agent 的一个简单定义：LLM 在循环中自主使用工具。

与客户合作的过程中，我们看到该领域正在向这个简单范式收敛。随着底层模型变得更加强大，Agent 的自主程度可以扩展：更智能的模型允许 Agent 独立导航细微的问题空间并从错误中恢复。

我们现在看到工程师在为 Agent 设计上下文的方式上发生了转变。如今，许多 AI 原生应用采用某种形式的基于 Embedding（嵌入）的推理前检索来为 Agent 提供重要上下文进行推理。随着该领域向更具 Agent 性的方法过渡，我们越来越多地看到团队用"即时"上下文策略来增强这些检索系统。

采用"即时"方法构建的 Agent 不会预先处理所有相关数据，而是维护轻量级标识符（文件路径、存储的查询、网页链接等），并使用这些引用在运行时通过工具动态加载数据到上下文中。Anthropic 的 Agent 编程解决方案 Claude Code 就使用这种方法对大型数据库执行复杂的数据分析。模型可以编写有针对性的查询、存储结果，并利用 Bash 命令如 head 和 tail 来分析大量数据，而无需将完整的数据对象加载到上下文中。这种方法反映了人类认知：我们通常不会记住整个信息语料库，而是引入外部组织和索引系统（如文件系统、收件箱和书签）来按需检索相关信息。

💡 举例说明：想象你在做一个大型代码库的迁移任务。与其把整个代码库加载到上下文中（会消耗大量 Token），不如让 Agent 先通过 glob 查看文件结构，再通过 grep 搜索特定模式，最后只读取需要修改的文件。这就是"即时"上下文策略的实际应用。

除了存储效率之外，这些引用的元数据还提供了一种高效精炼行为的机制，无论是显式提供还是直觉性的。对于在文件系统中操作的 Agent 来说，tests 文件夹中名为 test_utils.py 的文件与位于 src/core_logic/ 中的同名文件暗示着不同的用途。文件夹层次结构、命名约定和时间戳都提供了重要信号，帮助人类和 Agent 理解如何以及何时利用信息。

让 Agent 自主导航和检索数据还实现了 Progressive Disclosure（渐进式披露）——换句话说，允许 Agent 通过探索逐步发现相关上下文。每次交互都会产生为下一个决策提供信息的上下文：文件大小暗示复杂性；命名约定暗示用途；时间戳可以作为相关性的代理。Agent 可以逐层构建理解，只在工作记忆中保留必要的内容，并利用笔记策略实现额外的持久性。这种自我管理的上下文窗口使 Agent 专注于相关子集，而不是淹没在详尽但可能无关的信息中。

💡 术语解释：Progressive Disclosure（渐进式披露）是一种交互设计模式，指逐步展示信息而非一次性全部呈现。在 Agent 上下文中，它意味着 Agent 通过探索逐步发现和加载相关上下文，而非预先加载所有信息。

当然，这里存在一个权衡：运行时探索比检索预计算数据更慢。不仅如此，还需要有主见和深思熟虑的工程来确保 LLM 拥有正确的工具和启发式方法来有效导航其信息环境。如果没有适当的指导，Agent 可能会因误用工具、追逐死胡同或未能识别关键信息而浪费上下文。

在某些场景下，最高效的 Agent 可能采用混合策略，预先检索一些数据以提高速度，然后根据其判断进行进一步的自主探索。"正确"自主水平的决策边界取决于任务。Claude Code 就是一个采用这种混合模型的 Agent：CLAUDE.md 文件被预先加载到上下文中，而 glob 和 grep 等原语允许它导航环境并即时检索文件，有效地绕过了索引过时和复杂语法树的问题。

混合策略可能更适合内容变化较少的上下文，如法律或金融工作。随着模型能力的提升，Agent 设计将趋向于让智能模型智能地行动，人类策展工作逐渐减少。鉴于该领域快速发展的步伐，"做最简单有效的事情"可能仍然是我们对在 Claude 上构建 Agent 的团队的最佳建议。

### 长期任务的上下文工程

长期任务要求 Agent 在 Token 数量超过 LLM 上下文窗口的动作序列中保持连贯性、上下文和目标导向的行为。对于跨越数十分钟到数小时持续工作的任务，如大型代码库迁移或综合研究项目，Agent 需要专门的技术来绕过上下文窗口大小限制。

等待更大的上下文窗口似乎是一个显而易见的策略。但很可能在可预见的未来，所有大小的上下文窗口都会受到上下文污染和信息相关性问题的影响——至少在需要最强 Agent 性能的情况下是这样。为了使 Agent 能够在扩展的时间范围内有效工作，我们开发了几种直接解决这些上下文污染约束的技术：Compaction（压缩）、结构化笔记和多 Agent 架构。

#### Compaction（压缩）

Compaction 是一种实践，当对话接近上下文窗口限制时，对其内容进行摘要，然后用摘要重新启动一个新的上下文窗口。Compaction 通常作为上下文工程中驱动更好长期连贯性的第一个杠杆。在 Claude Code 中，模型会保留架构决策、未解决的 Bug 和实现细节，同时丢弃冗余的工具输出。Agent 以压缩的上下文加上最近访问的五个文件继续工作。

💡 举例说明：假设你让 Agent 帮你重构一个大型项目。经过几十轮对话后，上下文快满了。Compaction 会将之前讨论的关键决策、已完成的工作和待办事项压缩成一份摘要，然后用这份摘要开始新的对话，让 Agent 能够继续工作而不丢失关键信息。

最佳实践：首先最大化摘要中的召回率（确保不遗漏重要信息），然后迭代提高精确度（去除不相关细节）。

#### 结构化笔记

结构化笔记是指 Agent 定期将笔记持久化到外部记忆中。Anthropic 用 Claude 玩宝可梦（Pokémon）演示了这一点，Agent 在数千个游戏步骤中维护了精确的计数，跟踪诸如"在过去的 1,234 步中我一直在 Route 1 训练我的宝可梦，皮卡丘已经升了 8 级，目标是 10 级"等目标。一个基于文件的记忆工具已在 Claude 开发者平台上作为 Sonnet 4.5 发布的一部分进入公开 Beta。

这种策略以最小的开销提供了持久记忆。就像 Claude Code 创建待办事项列表，或者你的自定义 Agent 维护一个 NOTES.md 文件来跟踪进度和发现一样。

💡 术语解释：结构化笔记不同于简单的对话摘要。它是一种有组织的外部记忆系统，Agent 可以在需要时主动写入和读取，类似于人类在工作中记笔记并在需要时翻阅。

#### 多 Agent 架构

多 Agent 架构将专门的子 Agent 分配给具有干净上下文窗口的聚焦任务。主 Agent 协调高层计划，子 Agent 执行深入的技术工作。子 Agent 可以广泛探索（数万个 Token），但只向协调者返回 1,000-2,000 个 Token 的精简摘要。

💡 举例说明：假设你有一个研究任务需要分析 50 篇论文。主 Agent 可以将论文分成若干组，为每组启动一个子 Agent。每个子 Agent 深入阅读其分配的论文并生成一份精简摘要。主 Agent 然后综合所有摘要，生成最终的综合报告。这样，主 Agent 的上下文窗口只包含精简的摘要而非 50 篇论文的全文。

这种方法的优势在于：

  - 每个子 Agent 在干净、聚焦的上下文中工作

  - 主 Agent 不会被海量细节淹没

  - 可以并行处理多个子任务

  - 错误隔离——一个子 Agent 的失败不会影响其他子 Agent

### 结论

上下文工程代表了从编写提示词到管理整个上下文状态的范式转变。随着 Agent 变得更加自主和强大，精心策展上下文的能力将成为区分优秀 AI 工程师的关键技能。

核心原则始终不变：找到尽可能最小的、高信号 Token 集合，以最大化实现期望结果的可能性。无论是通过精心设计的系统提示词、最小可行工具集、即时上下文检索，还是长期任务管理技术，目标都是一样的——在有限的注意力预算下，让每一 Token 都发挥最大价值。

正如我们反复强调的，对于在 Claude 上构建 Agent 的团队来说，"做最简单有效的事情"仍然是我们最好的建议。
