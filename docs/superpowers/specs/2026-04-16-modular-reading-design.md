# 长文章模块化阅读设计方案

## 1. 项目背景

当前 `raw/articles` 目录下存在一些较长的文章，影响阅读体验和维护效率。通过分析发现，以下文章超过了 10,000 字符：

- `mcp-deep-dive.md`：19,114 字符
- `agent-framework-practice.md`：15,021 字符
- `swe-bench.md`：13,938 字符

这些长文章需要通过模块化拆分和双链连接来提高可读性和可维护性。

## 2. 现有系统分析

### 2.1 双链机制
- 使用 `[[wikilinks]]` 格式创建页面间链接
- 有 `layouts/shortcodes/wikilink.html` 处理双链渲染
- 支持 `[[link|display text]]` 格式的链接

### 2.2 项目结构
- `raw/articles/`：原始文章目录
- `concepts/`：概念页面目录
- `content/`：内容页面目录
- `entities/`：实体页面目录

### 2.3 现有规范
- 文件命名：小写，连字符分隔，无空格
- 页面结构：YAML frontmatter + 内容 + 相关链接
- 页面拆分阈值：超过 ~200 行的页面应拆分为子主题

## 3. 模块化方案

### 3.1 方案 1：基于章节的模块化

**描述**：将长文章按章节拆分为多个文件，主文件包含目录和概述。

**优点**：
- 结构清晰，符合原始文章的组织结构
- 导航方便，可通过双链快速跳转
- 维护成本低，修改特定章节只需更新对应文件

**缺点**：
- 文件数量增加
- 需要保持章节间的一致性

**实施步骤**：
1. 分析文章结构，识别主要章节
2. 创建主文件，包含目录和概述
3. 为每个章节创建单独的文件
4. 使用双链连接主文件和章节文件
5. 在章节文件中添加返回主文件的链接

### 3.2 方案 2：基于主题的模块化

**描述**：将长文章按主题拆分为多个文件，主文件包含主题索引。

**优点**：
- 主题集中，易于理解和查找
- 可根据主题相关性组织内容
- 适合内容较为分散的文章

**缺点**：
- 可能破坏原始文章的线性结构
- 主题边界可能不够明确

**实施步骤**：
1. 分析文章内容，识别主要主题
2. 创建主文件，包含主题索引
3. 为每个主题创建单独的文件
4. 使用双链连接主文件和主题文件
5. 在主题文件中添加相关主题的链接

### 3.3 方案 3：混合式模块化

**描述**：结合章节和主题的优点，根据文章特点灵活选择拆分方式。

**优点**：
- 灵活性高，适合各种类型的长文章
- 平衡了文件数量和内容组织
- 可根据内容重要性调整拆分粒度

**缺点**：
- 需要更多的规划和设计
- 一致性管理较为复杂

**实施步骤**：
1. 分析文章结构和内容
2. 确定主要章节和主题
3. 创建主文件，包含目录和概述
4. 为重要章节和主题创建单独的文件
5. 使用双链连接相关文件

## 4. 推荐方案

**推荐方案**：方案 1 - 基于章节的模块化

**推荐理由**：
- 符合现有项目的组织结构和规范
- 与原始文章的线性结构保持一致
- 实施简单，易于理解和维护
- 符合 SCHEMA.md 中的页面拆分原则

**放弃原因**：
- 方案 2 可能破坏原始文章的结构，不适合技术文档
- 方案 3 虽然灵活，但增加了实施复杂度，对于当前项目来说过于复杂

## 5. 具体实施细节

- 主文件：保持原文件名，如 `agent-framework-practice.md`

- 章节 slug：使用小写，连字符分隔，去除特殊字符

## 6. 实施计划

### 6.1 优先级

1. `mcp-deep-dive.md`（19,114 字符）
2. `agent-framework-practice.md`（15,021 字符）
3. `swe-bench.md`（13,938 字符）

### 6.2 步骤

1. **分析文章结构**：使用 `modularize.py` 脚本分析文章结构和章节
2. **创建主文件**：更新主文件，添加目录和概述
3. **创建章节文件**：为每个章节创建单独的文件
4. **添加双链**：在主文件和章节文件中添加相应的双链
5. **测试链接**：确保所有双链都能正确工作
6. **更新索引**：将新创建的文件添加到 `index.md`
7. **记录日志**：在 `log.md` 中记录模块化操作

## 7. 维护策略

### 7.1 更新机制

- 当原始文章更新时，同步更新主文件和相关章节文件
- 保持 `updated` 日期的一致性
- 使用相同的版本号标记相关文件

### 7.2 质量控制

- 定期检查双链的有效性
- 确保章节文件的内容与主文件的目录一致
- 保持文件命名和结构的一致性

### 7.3 文档规范

- 遵循现有项目的文档规范
- 保持 YAML frontmatter 的一致性
- 使用统一的双链格式和命名规范

## 8. 预期效果

通过模块化拆分和双链连接，预期达到以下效果：

- **提高可读性**：将长文章拆分为多个小模块，每个模块专注于一个主题
- **便于导航**：使用双链可以快速在不同模块之间跳转
- **易于维护**：修改特定内容时只需更新对应的模块文件
- **知识连接**：通过双链建立模块之间的关联，形成知识网络
- **减少认知负担**：读者可以根据需要选择性阅读特定模块

## 9. 风险评估

### 9.1 潜在风险

- **链接失效**：如果文件命名或路径发生变化，可能导致双链失效
- **内容不一致**：如果更新时没有同步所有相关文件，可能导致内容不一致
- **文件数量增加**：模块化会增加文件数量，可能影响项目的管理

### 9.2 风险缓解

- **使用脚本**：使用脚本自动生成和检查双链
- **统一更新**：建立统一的更新机制，确保相关文件同步更新
- **文件组织**：合理组织文件结构，使用目录来管理相关文件

## 10. 结论

通过基于章节的模块化方案，可以有效提高长文章的可读性和可维护性。该方案符合现有项目的组织结构和规范，实施简单，易于理解和维护。通过双链连接，可以建立模块之间的关联，形成一个结构化的知识体系，为读者提供更好的阅读体验。
