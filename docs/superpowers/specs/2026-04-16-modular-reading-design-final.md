# 长文章模块化阅读设计方案（最终版）

## 1. 项目背景

当前 `raw/articles` 目录下存在一些较长的文章，影响阅读体验和维护效率。通过分析发现，以下文章超过了 10,000 字符：

- `mcp-deep-dive.md`：19,114 字符
- `agent-framework-practice.md`：15,021 字符
- `swe-bench.md`：13,938 字符

这些长文章需要通过模块化拆分和双链连接来提高可读性和可维护性，同时确保**原文仍然维护在 raw 文件夹下**，不影响现有的维护结构。

## 2. 现有系统分析

### 2.1 项目结构
- `raw/articles/`：原始文章目录（从外部导入的原始内容，需要保持不变）
- `concepts/`：概念页面目录（处理后的 wiki 页面）
- `content/`：内容页面目录（Hugo 站点内容）
- `entities/`：实体页面目录
- `comparisons/`：比较页面目录
- `queries/`：查询页面目录

### 2.2 维护工作流
- **内容来源**：从 Notion 导入原始内容到 `raw/articles/`
- **处理流程**：通过 `importer.py` 工具处理原始内容，生成 wiki 页面
- **发布流程**：使用 Hugo 生成静态网站
- **索引管理**：通过 `index.md` 维护页面索引
- **日志管理**：通过 `log.md` 记录维护操作

### 2.3 双链机制
- 使用 `[[wikilinks]]` 格式创建页面间链接
- 有 `layouts/shortcodes/wikilink.html` 处理双链渲染
- 支持 `[[link|display text]]` 格式的链接

### 2.4 现有规范
- 文件命名：小写，连字符分隔，无空格
- 页面结构：YAML frontmatter + 内容 + 相关链接
- 页面拆分阈值：超过 ~200 行的页面应拆分为子主题

## 3. 模块化方案

### 3.1 方案 1：创建模块化版本，保留原文（推荐）

**描述**：在 `raw/articles/` 目录下创建模块化版本的文件，同时保留原文不变。模块化文件使用与原文相同的命名前缀，添加 `-modular` 后缀或使用子目录。

**优点**：
- **保持原文不变**：原文文件完全保持原样，不影响现有维护工作流
- **模块化阅读**：通过模块化文件实现更好的阅读体验
- **兼容性好**：不影响现有的 `importer.py` 工具和其他维护流程
- **灵活度高**：可以根据需要调整模块化程度

**缺点**：
- 文件数量增加
- 需要保持模块化版本与原文的同步

**实施步骤**：
1. 分析文章结构，识别主要章节
2. 保留原文文件不变
3. 创建模块化主文件，包含目录和概述
4. 为每个章节创建单独的文件
5. 使用双链连接主文件和章节文件
6. 在章节文件中添加返回主文件的链接

### 3.2 方案 2：使用符号链接

**描述**：保留原文文件不变，创建模块化文件，通过符号链接指向原文的不同部分。

**优点**：
- 保持原文不变
- 节省存储空间
- 模块化阅读体验

**缺点**：
- 符号链接在不同操作系统间可能存在兼容性问题
- 实现复杂度较高
- 可能影响版本控制系统

**实施步骤**：
1. 分析文章结构，识别主要章节
2. 保留原文文件不变
3. 创建模块化主文件，包含目录和概述
4. 为每个章节创建符号链接，指向原文的相应部分
5. 使用双链连接主文件和章节链接

### 3.3 方案 3：使用前端渲染

**描述**：保持原文文件不变，通过 Hugo 短代码和前端技术实现模块化导航。

**优点**：
- 保持原文不变
- 不需要创建额外文件
- 维护成本低

**缺点**：
- 依赖 Hugo 渲染能力，本地阅读体验不佳
- 无法真正实现文件级别的模块化
- 对长文章的编辑和维护仍然困难

**实施步骤**：
1. 在长文章中添加章节标记
2. 创建 Hugo 短代码实现章节导航
3. 在渲染时生成模块化导航界面

## 4. 推荐方案

**推荐方案**：方案 1 - 创建模块化版本，保留原文

**推荐理由**：
- **保持原文不变**：原文文件完全保持原样，确保现有的维护工作流不受影响
- **模块化阅读**：通过创建专门的模块化文件，实现更好的阅读体验
- **兼容性好**：不影响现有的 `importer.py` 工具和其他维护流程
- **实施简单**：只需添加新文件，不需要修改现有工具和流程
- **灵活度高**：可以根据需要调整模块化程度

**放弃原因**：
- 方案 2 实现复杂度较高，且符号链接可能存在兼容性问题
- 方案 3 虽然维护成本最低，但无法真正实现文件级别的模块化，对长文章的编辑和维护仍然困难

## 8. 预期效果

通过创建模块化版本，同时保留原文不变，预期达到以下效果：

- **保持原文完整性**：原文文件完全保持原样，确保现有的维护工作流不受影响
- **提高可读性**：将长文章拆分为多个小模块，每个模块专注于一个主题
- **便于导航**：使用双链可以快速在不同模块之间跳转
- **易于维护**：修改特定内容时只需更新对应的模块文件
- **知识连接**：通过双链建立模块之间的关联，形成知识网络
- **减少认知负担**：读者可以根据需要选择性阅读特定模块
- **保持兼容性**：不影响现有的 llm wiki 维护结构和工作流

## 9. 风险评估

### 9.1 潜在风险

- **内容不同步**：如果原文更新时没有同步更新模块化版本，可能导致内容不一致
- **文件数量增加**：模块化会增加文件数量，可能影响项目的管理
- **维护成本增加**：需要同时维护原文和模块化版本

### 9.2 风险缓解

- **使用脚本**：使用脚本自动生成和同步模块化版本
- **统一更新**：建立统一的更新机制，确保相关文件同步更新
- **保持结构**：所有文件保持在 `raw/articles/` 目录下，不改变现有结构
- **测试验证**：在实施后测试所有链接和导入流程

## 10. 结论

通过创建模块化版本，同时保留原文不变的方案，可以有效提高长文章的可读性和可维护性，同时确保现有的 llm wiki 维护结构不受影响。该方案符合现有项目的组织结构和规范，实施简单，易于理解和维护。

该方案的关键优势在于：
- **保持原文不变**：确保现有的维护工作流和工具不受影响
- **模块化阅读**：通过创建专门的模块化文件，实现更好的阅读体验
- **高兼容性**：与现有的 `importer.py` 工具和 Hugo 渲染流程完全兼容
- **易实施性**：只需添加新文件，不需要修改现有工具和流程
- **可扩展性**：可以根据需要扩展到其他长文章

通过这种方式，可以在不影响现有维护结构的情况下，实现长文章的模块化阅读，提高整个 wiki 系统的质量和可用性。
